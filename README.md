## 项目概述

SkyCompass Satellite 是 SkyCompass 项目的扩展演化版本，运行在 M5Stack Cardputer ADV (ESP32) 上。与原项目关注自然天体（太阳、月亮等）不同，Satellite 版本专注于**人造飞行器（如国际空间站 ISS、天宫空间站、哈勃望远镜等）**的实时轨道追踪与可视性预测。

**核心产品定位：** 向用户回答：**“近期有哪些人造飞行器值得观测？”**，它不仅是一个轨道展示工具，更是一个“今晚值得抬头看的飞行器推荐系统”。

## 硬件支持

- M5 Cardputer ADV（ESP32）
- 六轴IMU（加速度计 + 陀螺仪）
- 内置GNSS模块（用于定位及获取高精度 UTC 时间）
- 键盘 + 小尺寸屏幕

## 核心特性

- **全景快捷交互与按键映射**：
  - `Enter`：展开/收起近期最佳观测窗口推荐面板。
  - `S` 键：调出卫星复选菜单，自由决定同屏显示的目标。菜单内按 `d` 键可删除自定义卫星。
  - `W` 键：开启/关闭 WiFi 并进行 NTP 时间同步与轨道数据更新。
  - `H` 键：随时在屏幕中央悬浮呼出**快捷键帮助菜单**。
  - `C` 键：进入/退出**手动定位模式 (十字准星模式)**，使用 `;`(上) `.`(下) `,`(左) `/`(右) 键如同操纵摇杆般自由拨动地球，手动指定任意观测地点。
  - `G` 键：开启/关闭 **GNSS 强制常开模式**，退出系统自动休眠策略以手动获取定位。
  - `V` 键：进入/退出**飞行器跟随视角 (Sat View Mode)**，在此模式下可通过 `;`(上一个) 和 `.`(下一个) 键快速切换当前聚焦的卫星。
  - `Space` 键 (空格)：开启/关闭 **IMU 视角锁定**。在跟随视角中，按空格可锁定当前观察姿态，释放后视角将重新跟随设备的物理转动。
  - `R` 键：一键重置**时光机**模拟时间，立刻恢复至实际系统时间。
  - `Del` 键 (Backspace)：开启/关闭 **四角数据 HUD 叠加显示**，隐藏角落的数据。
  - `[` 和 `]` 键：在主界面调整屏幕亮度（最低支持16，最高255）。
- **实时模式 (Live Mode) & 时光机 (Time Machine)**：
  - 纯代码绘制极简地球与**智能冷暖色渐变大陆轮廓**，动态渲染昼夜交替。
  - 具备**视觉锚点系统**：自动测算并在屏幕角落浮现“动态指北针”；在极点区域渲染“十字虚线与立体地轴”，彻底解决上帝视角下的方向迷失问题。
  - 按下 **`,` (后退)** 或 **`/` (快进)** 键可以呼出**时光机**，支持长按极速穿梭时间，直观洞察未来的过境轨迹。当时间存在偏移时，右下角 HUD 时间将显示为**黄色**，以提供强烈的视觉反馈；在主界面按下 **`R`** 键可一键将模拟时间瞬间重置为实际现实时间（字体恢复为白色）。
  - **任务驱动型架构 (Mission-Driven Architecture)**：
  为了应对成组发射任务（如星链 20~30 颗卫星）可能导致 ESP32 内存耗尽的问题，Recent Launch 模块进行了重构设计，遵循 **Mission First** 的架构：
  - **三级导航架构**：系统包含 Mission 列表（展示近期发射）、Mission Overview（展示 Hover 任务的开普勒代表轨道参数、年龄、状态评分及可见性）以及 Objects 查看（按 `O` 键进入，仅在内存中懒加载当前页 5 颗卫星，退出或翻页时即刻完全释放 SGP4 实例）。
  - **基于真实轨道相位的任务编队可视化 (Mission Formation Visualization)**：在不增加 SGP4 计算负担的前提下，利用真实轨道数据还原星座空间分布。系统解析全组卫星的 Mean Anomaly（平近点角）参数作为 AlongTrackPhase（沿轨相位），在加载时通过**环状层次聚类算法**（Agglomerative Clustering）将原始大数量卫星压缩为 5~8 个 Proxy 卫星（代理卫星），并在代表轨道上通过线性插值进行投影绘制。同时，系统通过寻找环上最大 Gap（间隙）自适应计算出编队在轨道上占据的真实物理度数（Occupancy），并在地球仪轨道上实时渲染出一段发光带。在 CPU 与内存开销几乎为零的前提下，呈现高真实度、且在时间穿梭中完全连贯移动的星链列车演化态势。
  - **NASA 卫星夜间灯光与光污染点云叠加**：自动从 NASA GIBS 官方服务中抓取全球最新的 VIIRS Black Marble 瓦片数据，利用重要性采样算法生成 2000 个高对比度的夜景光害亮点（已通过极地地理坐标约束彻底排除了极地冰盖与瓦片边界水印的反射噪声）。在设备端的黑夜区域，通过**内联正交投影与快速背面剔除**算法以 30 FPS 进行流畅的 3D 渲染，并结合晨昏线进行**“平滑黄昏渐变”**绘制（日落时星光亮起，深夜达到满值，拂晓熄灭）。
- **观测推荐系统 (核心价值)**：
  - 在主界面按 **`Enter` 键** 可滑出左侧推荐面板。
  - 综合计算地球遮挡、太阳照明，结合亮度评分输出最佳观测窗口（AOS、最高仰角时间）。
- **高保真 RGB565 原始像素截图传输系统**：
  - 针对 Cardputer **无 PSRAM** 的硬件硬伤，绕过耗费内存的 PNG 压缩，首创**零压缩原始像素串口传输机制**。
  - 设备端通过按下侧面按钮（G0）触发，直接将画布（Sprite）中 64KB 的原始 RGB565 像素分段 Base64 编码，并通过魔术前缀 `==SKYCOMPASS_DATA==` 发往默认的 stdout 串口日志通道。
  - 电脑端配备纯 Python 开发的截图监听脚本（`scripts/get_screenshot.py`，兼容 GBK 编码环境），能自动搜索设备串口，动态剔除 ANSI 颜色控制字符和并发的多线程杂余日志干扰，完美还原并保存为高保真 24 位 BMP 图片。
- **设备端全闭环运行与离线生存能力**：
  - 支持按下 **`W` 键** 直接在设备屏幕上扫描周边热点并使用全键盘输入密码连接。
  - 连网后全自动完成 NTP 时间同步、Celestrak 最新 TLE 轨道根数下载（写入本地 LittleFS 闪存）以及 **IP 城市级快速定位**。
  - **离线秒开与 GNSS 同步**：无网环境下，开机秒速加载本地 LittleFS 闪存的轨道缓存；一旦在室外获得 GNSS 锁定，设备会自动解析原子钟 UTC 时间校准推演时间轴，并更新定位至真实坐标。
  - **智能刷新策略 (48小时阈值)**：TLE 数据有效期被设定为 48 小时。网络任务仅在缓存超过此阈值时才会重新下载，完美兼顾肉眼/望远镜观测对时效性的不同要求。
- **极致的电源管理**：
  - 网络任务处理完毕后，立刻自动切断 WiFi 射频以节省电量。
  - 对于 GNSS 定位模块，一旦成功获得 3D Fix 并在同步时空坐标后，或搜星超时（1分钟）未果，即向其下发休眠指令（`$PCAS10,0*1C`），大幅提升续航表现。
- **云端无线电频率下发 (API Gateway)**：为了解决单片机内存极小无法解析超大官方频率 JSON 的问题，本项目采用了 GitHub 仓库作为云端配置下发源。
  - 维护本仓库中的 `data/frequencies.json`，并利用自带的 `.github/workflows/update_frequencies.yml` 自动化流水线。
  - GitHub 会定期在云端执行庞大的 API 过滤任务，生成轻量级 JSON 喂给设备。设备连网后自动拉取，实现无缝的 HAM 通联频率与自定义卫星配置自动同步！
- **定制化无线电与卫星分类 UI (Tailored Satellite Type UI)**：
  根据卫星本身的属性与无线电功能，系统将所有卫星分为 5 类，并在百科面板、跟随视角 HUD、侧栏局部 HUD 中展示定制的界面布局，杜绝了猜测或生成虚构无线电频率的可能：
  - **HAM (业余无线电卫星)**：展示 Rx（经实时多普勒补偿）、Tx（发射上行频率及哑音 Tone）及通信模式，非追踪状态下智能分析并展示下一次过境的 AOS、LOS 和最高仰角 (Max El)。
  - **WEATHER (气象卫星)**：如 NOAA、Meteor 系列，仅渲染 Rx 下行多普勒接收频率与气象成像模式（如 APT/LRPT），屏蔽无用的 Tx 上行参数。
  - **SPACE_STATION (空间站)**：针对国际空间站 (ISS) 的复杂无线电载荷，定制同屏展示 APRS (145.825 MHz) 与 SSTV (145.800 MHz) 双通道的实时多普勒频移；对于天宫等无 HAM 活动的空间站则显示无无线电能力。
  - **VISUAL (仅目视观测卫星)**：如哈勃望远镜、JWST 以及各类火箭残骸，UI 统一明确显示 `No Amateur Radio Capability`，绝不虚构任何无线电数据。
  - **HISTORICAL (历史失效卫星)**：如东方红一号 (DFH-1)，展示 1970 年发射的背景与已失效 (Inactive) 状态，且在追踪/非追踪状态下均不绘制任何频率栏，保留历史致敬价值。


## 轨道推演与可见性预测技术细节

为了保证与专业天文软件的预测结果高精度对齐，本项目在嵌入式端（ESP32）实现了一套完整的空间坐标转换、天体力学与光学可视度推演算法。以下是各项计算的核心物理模型和要点描述：

### 1. 时间系统与恒星时转换
* **儒略日 (Julian Date)**
  系统利用本地获取的 GPS/NTP 高精度 UTC 时间戳，转换成天文学轨道解算所需的儒略日（Julian Date），为后续 SGP4 模型的计算提供高精度时间基准。
* **格林尼治平恒星时 (GMST)**
  由于地球自转，在进行惯性坐标系到地球固定坐标系的转换时，必须实时计算格林尼治平恒星时角。算法基于 IAU 天文学标准公式，根据当前的儒略日计算出地球当前的自转角度，最终转换并取模为 [0, 2π] 的自转自变量弧度值。

### 2. 空间坐标系变换 (TEME -> ECEF -> ENU)
* **TEME 惯性坐标系到地固坐标系 (ECEF)**
  SGP4 轨道模型直接输出的卫星位置矢量处于 TEME 惯性坐标系中。项目使用格林尼治平恒星时（GMST）作为旋转角构建旋转矩阵，将卫星的位置坐标投影到地球固定坐标系（ECEF）中，从而把惯性空间位置转换为地球表面的三维地固坐标。
* **WGS-84 地球椭球体参数**
  考虑到地球并非完美的球体，算法采用 WGS-84 地球模型（引入赤道半径与第一偏心率参数）进行精确投影。
  - **大地坐标转换为 ECEF**：将观测者（用户）的经度、纬度及实时海拔高度（自动从 GPS 或网络定位中获取）转换为三维 ECEF 地固坐标。
  - **ECEF 转换为大地坐标**：使用带有高精度循环的迭代解算，修正椭球体形状带来的纬度/海拔畸变，用以准确算出卫星星下点的地理位置。
* **站心坐标系 (Topocentric ENU, East-North-Up)**
  为了获取卫星相对于地面观测者的观测视角，计算卫星与观测者之间的三维相对距离矢量，并通过观测者切平面旋转矩阵投影到东北天（ENU）坐标系中。以此最终解算出卫星实时的方位角（Azimuth）、经过地球曲率修正的仰角（Elevation）以及距离观测者的视距（Range）。

### 3. 大气折射修正 (Atmospheric Refraction)
当卫星仰角较低时，地球大气层会对光线产生明显的折射作用，使得其实际视觉仰角偏高。本项目在低高度角区间内引入了大气折射角修正。该修正显著提升了低仰角过境的出地平（AOS）和入地平（LOS）时间预测精度，使其与 Heavens-Above 等专业天文预测软件保持高度吻合。

### 4. 严格光学可视性三要素
满足“肉眼可见”的人造天体必须严格同时满足以下三个物理条件：
1. **仰角阈值限制**：卫星高度角满足大于等于设定好的高度角阈值（通常为 10.0°），以避开地面建筑物与近地表低空大气的光学遮挡。
2. **观测者夜间环境**：计算当前时刻太阳相对于观测者的局部仰角。只有当太阳高度降至民用暮光以下（即太阳高度角小于 -6.0°，天空完全变暗）时才判定为黑夜环境。
3. **本影锥阴影判定 (Earth Umbra Check)**：卫星本身必须被阳光照射。算法采用地球本影锥近似球体阴影模型，通过计算太阳直射点与卫星的夹角，判断卫星是否没入地球阴影区。若卫星处于地球本影中，则阳光无法照射，呈不可见状态。

### 5. 视星等、漫反射相函数与大气消光修正 (Magnitude, Phase Angle & Atmospheric Extinction)
为了评估卫星过境时的真实肉眼可视亮度，算法结合了卫星标准星等、实时视距、太阳光照相位角以及大气消光项：
* **相位角 (ψ)**：计算“太阳-卫星-观测者”三维夹角，用以确定反射阳光的几何关系。
* **漫反射球体相函数**：使用球体漫反射模型来描述太阳光在卫星机体表面发生漫反射后的光强空间分布。
* **大气消光修正**：引入 Kasten-Young 大气质量模型与视觉波段消光系数，用于修正低高度角时由于穿透大气层较厚引起的光强吸收和散射衰减。
* **视星等合成**：综合视距、相位角和低仰角大气消光项，计算得出精准的实时视星等，使亮度预测结果与专业天文预报软件深度对齐。

### 6. 高能效双重推演步长机制与时光机滑动性能优化
针对 ESP32 运算性能受限的痛点，推荐与主渲染系统采用了一系列能效优化算法：
* **双重步长推演**：预推演阶段先采用 120 秒（2分钟）的大步长快速越过无过境的漫长时间段。一旦检测到卫星仰角跨越地平线，推演引擎立刻**将时间轴向后回滚 120 秒**，并切换为 **10 秒精细步长**进行高精度轨道求解，以锁定最精确的 AOS/最大仰角时刻/可视亮度及 LOS 时刻。过境事件结束后重新恢复大步长，完美兼顾了高计算精度与极低的单片机 CPU 开销。
* **物理时间计算冷却门限（Scrubbing Cooldown Gate）**：为了防止在通过“时光机”快进/快退或短按点按键滑动时间轴时，由于时间坐标频繁且连续改变而触发多颗卫星 3D 轨道线条的密集重算（每颗卫星需推演前后 45 分钟共 30 个点的 SGP4 坐标与大地坐标迭代），我们在 `calculateOrbit` 中引入了 **120 毫秒物理时间冷却保护**。在时间连续变动时，物理时间 120ms 内强制拦截并拒绝任何轨道的二次重算，只保留最核心的当前坐标更新；一旦按键松开或停止滑动，轨道线将在 120ms 后顺畅重绘。这彻底消除了长按 `,` 或 `/` 时的 CPU 占用尖峰（Lag Spike），将时间滑动时的渲染帧率提升至丝滑的 **60 FPS**。

### 7. 业余无线电 (HAM) 通联参数与多普勒频移计算
对于支持业余无线电（HAM）功能的卫星（如 FM 转发器卫星 SO-50、AO-91，或搭载多重无线电载荷的国际空间站 ISS），系统在百科页面及跟随视角 HUD 中展示了实时的通联频率参数。这些参数的物理含义与数学计算如下：

* **RX (Receive - 下行接收频率)**：卫星向地面发送信号的频率。程序中显示的 RX 频率是**实时经过多普勒频移补偿计算后**的实际监听频率。
* **TX (Transmit - 上行发射频率)**：地面电台向卫星发送信号的频率。在 UI 中通常显示其静态中心频率（在实际通联中，地面站需根据多普勒频移反向微调发射频率）。
* **T (Tone - 亚音频/哑音)**：即 CTCSS（Continuous Tone-Coded Squelch System，连续单音编码静噪系统）。在 UI 中以 `T: [Hz]`（例如 `T:67.0`）表示。它是开启卫星 FM 转发器所需的控制音频。如果未在发射信号中叠加此特定频率的亚音频，卫星的中继将不会被激活。若显示为 `None`，则表示不需要亚音频。
* **多普勒偏差值（括号中的数字）**：在 RX 频率右侧的括号中（例如 `(+1.5k)` 或 `(-3.2k)`）表示当前的**实时多普勒频移量 $\Delta f$（单位：kHz）**。
  * **多普勒频移物理模型**：
    由于卫星以极高速度（第一宇宙速度，约 7.8 km/s）相对于地面观测者运动，观测者接收到的无线电频率会发生偏移。算法根据卫星无线电中心频率、径向运动速度与光速的关系，实时测算多普勒频移量。卫星靠近观测者时频率升高，远离观测者时频率降低。
* **通联指导意义**：
    * **刚出地平线 (AOS)**：卫星快速靠近，多普勒频移达到正向最大值（如 `+3.5k`），电台需往高频调谐监听。
    * **过境天顶 (TCA / Max El)**：卫星相对径向速度为 0，频移为 0，此时接收频率等于卫星真实中心频率。
    * **入地平线 (LOS)**：卫星快速远离，多普勒频移达到负向最大值（如 `-3.5k`），电台需往低频调谐监听。
* **RX1 / RX2 与 U / D 的区别**：
  * 常规业余无线电中，**U (Uplink)** 代表上行频率，**D (Downlink)** 代表下行频率。
  * 针对具有复杂无线电载荷的多通道飞行器（如国际空间站 ISS），它同屏拥有多个不同的下行接收信道（例如 145.825 MHz 的 APRS 自动报位数据链，以及 145.800 MHz 的 SSTV 慢扫描电视图像传输）。为了充分利用屏幕空间，系统会将它们分别标注为 **RX1** 和 **RX2** 并排渲染，从而让无线电爱好者能够同时掌握双通道的多普勒频移补偿值，而非常规的单上单下（U/D）布局。

## 辅助工具与构建脚本

为了减轻 ESP32 在受限硬件资源下的运行时开销，以及方便用户进行数据交互，本项目在 `scripts` 目录下配备了多个离线数据预处理及电脑端配套工具：

1. **[gen_light_points.py](scripts/gen_light_points.py)** (光污染点云生成脚本)
   * **用途**：从 NASA GIBS 官方服务中下载最新的 VIIRS Black Marble 全球夜景瓦片地图并拼合为大图，通过分层重要性采样（Stratified Importance Sampling）提取出 3000 个高对比度的夜景光害地标点。
   * **输出**：自动计算出各坐标的球面三角函数并生成 [light_points_data.h](src/core/light_points_data.h)，供设备端直接免计算进行内联 3D 快速渲染。

2. **[generate_timezone_grid.py](scripts/generate_timezone_grid.py)** (离线时区网格生成脚本)
   * **用途**：生成全球 $1^\circ \times 1^\circ$ 精度的离线时区映射网格（180 行 × 360 列）。针对没有国家时区覆盖的公海等地区，采用航海时区自动兜底计算。
   * **输出**：生成 [timezone_grid.h](src/core/timezone_grid.h)，使设备在仅有 GPS 信号无网环境下，也能在秒级内获取当前所在位置的正确本地时间偏移量（UTC Offset）。

3. **[get_screenshot.py](scripts/get_screenshot.py)** (PC 端高保真截图监听脚本)
   * **用途**：电脑端运行的 Python 串口监听服务。自动扫描并绑定 Cardputer 的串口通道。在不破坏、不占用正常日志输出的前提下，利用正则和数据帧拦截技术，实时提取设备发回的 Base64 分段像素流。
   * **效果**：将 16 位大端 RGB565 原始帧缓冲区数据无损重构并保存为 24 位高保真的 BMP 图片，绕过了单片机内存不足以进行 PNG/JPG 压缩的硬件缺陷。

4. **[optimize_earth_data.py](scripts/optimize_earth_data.py)** (地表大陆线条离线优化脚本)
   * **用途**：将项目原版的浮点大陆轮廓经纬度线段数据进行离线三角函数预计算（计算 $\sin Lat, \cos Lat, \sin Lon, \cos Lon$ 等）。
   * **输出**：优化重构 [earth_data.h](src/core/earth_data.h)，省去单片机在 3D 转换矩阵中对每个地表点重复调用 `sin`/`cos` 的巨大计算开销，使地球线框旋转渲染流畅度稳定在 30 FPS。

5. **[update_frequencies.py](scripts/update_frequencies.py)** (业余无线电频率自动获取脚本)
   * **用途**：用于 GitHub Actions 自动化流水线。定期或在触发时从卫星通联数据库中拉取最新的 NORAD 卫星下行通联频率和调制模式，过滤并保持轻量化。
   * **输出**：更新生成本仓库的 `data/frequencies.json`，并通过 GitHub Pages 发布，供设备联网时一键更新。

6. **[scratch/](scratch)** 目录 (开发调试与仿真演练脚本)
   除了最终部署的辅助工具，本项目在开发过程中，于 `scratch/` 目录下编写并沉淀了大量用于算法原型验证、数值仿真校准以及代码合并过程中的临时辅助脚本。按用途分类梳理如下：

     * [benchmark.cpp](scratch/benchmark.cpp) / [test5.cpp](scratch/test5.cpp)：对浮点数矩阵乘法、三角函数公式等关键渲染逻辑在单片机上进行局部跑分与开销评估 of 的源码。

## 数据层架构 (Data Layer Architecture)

为了兼容 CelesTrak 自 2026 年起引入的 6 位 Catalog Number（100000+）且不再提供传统 TLE 格式的改变，本项目重构了数据层，实现了数据格式与轨道物理计算的完全解耦：

### 架构设计

```text
Network / WIFI
      │
      ▼
OrbitDataProvider
      │
      ▼
OrbitParser (TLEParser / JSONParser)
      │
      ▼
OrbitRecord (唯一可信数据对象 / DTO)
      │
      ▼
SGP4 (物理计算引擎)
      │
      ▼
Application (Earth View / Recommended / HAM / GUI)
```

### 支持的数据格式 (Supported Formats)

* **TLE (Legacy)**：传统双行元素法，自动兼容已有的历史卫星本地缓存。
* **CelesTrak GP JSON (Default)**：新一代默认数据交换格式，支持直接拉取 OMM 参数，原生支持 6 位 Catalog Number。
* **OMM (Reserved)**：CCSDS 轨道参数电文，已预留扩展接口。

整个系统已经兼容未来六位 Catalog Number，并使用轨道要素逆向构建 Pseudo TLE 的桥接技术，无缝兼容只接收 TLE 的底层 SGP4 物理库，同时保证业务展示层和数据存储层均保存完整、真实的 6 位 NORAD 卫星 ID。

## 高保真屏幕截图功能指南

针对 M5 Cardputer 无 PSRAM 的硬件极限，本项目采用了零内存、无压缩的 RGB565 串口像素流截图系统，并配备了 PC 端解码监听脚本。

### 1. 设备端触发方式
- 在任意主界面下，按下 Cardputer 侧面的 **BtnA（即 GPIO0 物理按键，位于开关右侧）**。
- 屏幕右上角会浮现 `Capturing screen...` 提示，此时设备将分段读取帧缓冲区并以 Base64 编码方式通过 USB 串口输出原始 RGB565 像素数据。

### 2. 电脑端接收方式
在电脑端，需要运行配套的 Python 脚本来监听并无损还原截图：

1. **安装环境依赖**：
   运行截图监听与转换需要 `pyserial` 和 `Pillow` 库：
   ```bash
   pip install pyserial Pillow
   ```
2. **运行监听脚本**：
   在项目根目录下，执行以下命令运行监听服务（它会自动扫描并寻找你的 Cardputer 串口设备）：
   ```bash
   python scripts/get_screenshot.py
   ```
3. **获取截图**：
   保持脚本运行，当您在 Cardputer 上按下截图键时，PC 端的 Python 终端会自动拦截 Base64 数据流并进行无损重构，截图将会自动保存为 24 位高保真的 `.bmp` 图片，存放于项目的 `screenshot/` 文件夹中。
